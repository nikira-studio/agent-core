# Integrations

Companion artifacts that help **other systems** integrate with Agent Core —
for cases that need more than an MCP connection and a prompt, or that materially
improve how reliably a host system uses Agent Core.

Most agents consume Agent Core two ways: connect it as an **MCP server**, and/or
paste one of the onboarding prompts generated by the dashboard Integrations page.
That's enough for the common case. This folder is for everything beyond that —
the glue that lives **inside another system** and talks to Agent Core over its
API/MCP.

## What belongs here

- **Plugins / providers** for host agent frameworks — e.g. a memory provider
  that prefetches Agent Core recall before every turn so the host agent can't
  answer from cold recall.
- **Skill files** that teach a host agent how to use Agent Core well.
- Helper **scripts, recipes, or config** for wiring a specific tool to Agent Core.

## What does NOT belong here

This is **not** part of the Agent Core runtime. Nothing here is imported by
`app/`, and it is excluded from the application test suite. It is also distinct
from Agent Core's own **adapter** system (`app/adapter_templates/`,
`data/adapters/`), which is how Agent Core calls *outbound* services — the
opposite direction from these integrations.

## Layout

```
integrations/
└── <system>/            # one folder per external system, e.g. hermes/
    └── <artifact>/      # the plugin / skill / script, each with its own README
```

## Why these live in the Agent Core repo

They depend on Agent Core's API and behavior, so they should be reviewed and
updated in the same place as the API they call — if an endpoint or schema
changes, the integration that relies on it changes in the same PR. Each
artifact's own README documents where it installs in the host system and how to
activate it.
